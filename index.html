记安装 unit 时引出的 python 安装问题

2018 年 11 月 16 日，在 nginx官网查看到 unit 服务器项目，想在自己的腾讯云主机上尝试安装。并配合unit-python使用

nginx/unit版本：1.6 python版本：Cpython 3.7.1

安装时遇到的问题

  1. 初期试错

    安装时遇到的问题在现在的主机不能重现，也就大致说一下。也就是提式动态库编译失败，建议添加-fPIC重新编译，我以为是现有的unit的 Makefile 没有添加这个编译选项，configure生成的 Makefile 也确实没有这个选项，然后我就添加了 python3:LDFLAGS += -fPIC -shared的选项以及后面干脆在全局添加 LDFLAGS= += fPIC -shared,结果还是不行，后来在网上查找解决办法发现，这个指的是原有动态库在编译时没有开启共享的选项，与正在编译的软件无关，也就是这里其实是Python3编译参数有关。

  2. 继续尝试

    重新 Python3 的 源码

    1. Python3 centos下的依赖

    zlib zlib-devel bzip2 bzip2-devel ncurses ncurses-devel readline readline-devel openssl openssl-devel openssl-static xz lzma xz-devel sqlite sqlite-devel gdbm gdbm-devel tk tk-devel libffi-devel

    （libffi-devel是3.7开始新增的依赖）

    1. 官网下载 Python3 的最新源码解压并进入解压的文件夹（安装autotools工具集和gcc编译器）
    2. 编译安装
    1. 配置PATH环境变量(~/.bashrc 或者~/.zshrc)。

    安装后就遇到其他问题，这时发现 Python3 解释器运行不了。

  3. 解决办法

    在 StockOverflow 上查到的解释大致是 Python3 在源码安装时虽然通过 -shared -fPIC 选项开启了共享库的选项，但因为安装在/usr/local/python3，没有指定共享库的真实路径，所以 Python 的执行文件无法找到这个共享库。

斐波拉契数引发的缓存优化问题

发表于 2018-10-01 | 更新于 2018-10-02

动机

​ 首先说说为什么写这篇文章，起因是在知乎上看到一篇关于一个面试者面试被面试官问到怎么去求斐波拉契数，继而问道怎么去优化求解过程，最后又被问到，如果服务器的空间有限，该怎么去优化这个过程，这其实和大学学的操作系统课程有点搭边了，可以用LRU去优化求解这个过程。

分析

什么是斐波拉契数列？

fib(n) = fib(n-1) + fib(n-2)

满足当前项为前两项的和的数列，第一、二项都为 1 的数列

可以怎么解决这个问题

从上面的表达式出发，这个问题其实是一个递归求和的问题的，可以用递归迭代求出第 n 项的值

这样做足够简单，但从中也看出问题

1、首先，我们看看为了解这样的问题我们需要做多少次运算

如果 value = 5

为了求 fib(5) 我们要求 fib(3) 和 fib(4)；

fib(3) 又要求 fib(1) 和 fib(2)；

fib(4) 要求 fib(2) 和 fib(3)；

这时求 fib(3) 其实是进入 fib(4) 的递归调用后的运算，所以必须重复执行；

所以这样做其实有很多重复的调用过程；

我们假设每次的递归时间开销相同，为k，那么解决这个问题的开销是 2^(n - 2)*k (n >=2)

2、Python解释器也是有递归次数限制的(通过sys.getrecursionlimit()查看， 我的机器上查看的结果为3000,儿在实际测试中，程序到达992次递归就抛出异常了)，学过C语言我们应该知道，C 语言每次递归时都要保存当前的堆栈状态，这样每次开销其实很大。

改进后其实还是没那么理想，如果进行随机的多次计算，很有可能之前计算过值短时间又会遇到，这个时候如果程序还是老老实实的计算，这样效率也太低了。

继续改进

这时如果计算的值的的范围比较固定，比如只有 1~500 的斐波拉契数的计算，这个时候这个函数效率表现挺好的

​ 如果我们这个时候遇到的不是求 斐波拉契数列的问题，结果范围很大，这个时候该怎么办呢，这个时候就引出我们要探讨的问题了, 我们需要在缓存结果很久未使用并且所设定的缓存空间满时将这些长时间没使用的数据清理掉。

​ 虽然长时间没使用的数据我们近期虽然不一定不会用，如果实际的应用场景并不适合用LRU去实现，可以考虑其他的实现方案。但始终记得的是，我们解决问题的方案要在较低的时间复杂度上去解决问题，不然，我们为了优化一类问题，却引入了另外一个拖慢运算的问题，这是得不偿失的。

后记

​ 除了自定义链表去容纳 LRU队列以外，还可以使用 heapq模块，因为堆的特点就是可以保持每次取出的结果是最小的，堆内存储的是 (次数, key) 组成的二元组，caches里存储 key、value键值对，用于判断存在性，命中后将次数加一，heapq.heappop操作一次。

​ 在 Python 中解决计算密集型任务还可以根据当前操作处理器的个数分配合理的进程数，将任务有效分配至多个进程上，进行完成。

Python学习之Mixin

发表于 2018-02-03 | 更新于 2018-02-04

写在前面

寒假在家准备写一个web版本的聊天软件，怎奈一直没有头绪，就花了时间去看django的源码，一直在别人代码里看到Mixin的字样，却不知道为什么这么写，心中的困惑总得不到消除，这次又看到了，所以索性到网上查了查，参考知乎上的解答。弄懂后记下。

概念

Mixin是一种特殊的多重继承，也就是多重继承的子集。
使用Mixin的好处是，同时享有单一继承的单纯性和多重继承的共有性。

作为Mixin类，需要满足以下条件：

  1. 不能单独生成实例对象，属于抽象类。
  2. 不能继承Mixin以外的类。

有以下特点

Mixin 类是单一职责的

Mixin 类对宿主类（Post）一无所知（等价于 Java 中要求宿主类实现 Entity 接口）

宿主类的主体逻辑不会因为去掉 Mixin 而受到影响，同时也不存在超类方法调用（super）以避免引入 MRO 查找顺序问题

自我见解

​ 通过我自己学的知识理解，也就是一种解耦合的过程，上层的方法提供者，也就是Mixin类提供了一系列可供子类调用的方法，而Mixin类不用知道调用它的子类是什么，比如threading.Thread类提供了一些线程操作的方法，而不用知道这些线程是用来完成什么事的，用socket.socket类配合可以实现多线程的socket通信。

​ Java对于接口有单独的类型interface去支持定义接口。

​ Python是支持多继承的语言，在学习C++的时候，我们就知道一件事情，如出现以下的情形，就麻烦了。

​ 使用Mixin，由于Mixin只关心它自己需要实现哪些方法供子类调用，且子类不要重写Mixin类中的同名方法，所以不会出现多重继承的弊病，比如下面的代码。

​ 这样设计的好处在于，即使不是转账不是由银行去使用，用于其他场景，也能运行的通，因为交易还是在账户和账户之间的，而不用关心到底调用的是商家还是银行，或许这和去年的一个比较流行的词汇有点像，“赋能”，这样做出一些Entity和Mixin，相互组合就可以完成不同场景的功能。

map/reduce

  1. map()

    传入的第一个参数是f，即函数对象本身。由于结果r是一个Iterator，Iterator是惰性序列，因此通过list()函数让它把整个序列都计算出来并返回一个list

  2. reduce()

    reduce把一个函数作用在一个序列[x1, x2, x3, …]上，这个函数必须接收两个参数，reduce把结果继续和序列的下一个元素做累积计算

filter()

和map()类似，filter()也接收一个函数和一个序列。和map()不同的是，filter()把传入的函数依次作用于每个元素，然后根据返回值是True还是False决定保留还是丢弃该元素
